# NEXTIME --- PRODUCT CHANGE REQUESTS

Version: 1.0 Status: Active Last Updated: 2026-07-21

# Purpose

This document defines the official process for requesting, evaluating,
approving, and tracking product changes within the NEXTIME platform.

Its objective is to ensure that every proposed change is documented,
evaluated consistently, and aligned with the product vision.

------------------------------------------------------------------------

# Change Request Status

Each Product Change Request (PCR) must have one of the following
statuses:

-   Proposed
-   Under Review
-   Approved
-   Rejected
-   Scheduled
-   In Progress
-   Implemented
-   Cancelled

------------------------------------------------------------------------

# Product Change Request Template

## PCR ID

PCR-XXX

### Title

Short and descriptive title.

### Date

YYYY-MM-DD

### Requested By

-   Customer
-   Product Team
-   Engineering
-   Support
-   Sales
-   Management

### Status

Current lifecycle status.

### Priority

-   Critical
-   High
-   Medium
-   Low

### Category

-   Feature
-   Enhancement
-   Bug Fix
-   UX Improvement
-   Performance
-   Security
-   Infrastructure
-   Documentation

------------------------------------------------------------------------

# Description

Describe the requested change clearly.

------------------------------------------------------------------------

# Business Justification

Explain why this change is valuable.

Possible benefits:

-   Better user experience
-   Increased productivity
-   Reduced support effort
-   Revenue growth
-   Competitive advantage

------------------------------------------------------------------------

# Impact Assessment

## Product Impact

Describe effects on workflows and usability.

## Technical Impact

Describe architectural implications, dependencies and complexity.

## Business Impact

Describe expected business outcomes.

------------------------------------------------------------------------

# Estimated Effort

Story Points or Engineering Days.

Dependencies should also be documented.

------------------------------------------------------------------------

# Risks

Identify implementation risks and mitigation actions.

------------------------------------------------------------------------

# Decision

Final decision should include:

-   Approved by
-   Approval date
-   Target release
-   Related Sprint
-   Linked Product Decision (PD)
-   Related ADR (if applicable)

------------------------------------------------------------------------

# Traceability

Each PCR should reference:

-   Product Requirements
-   User Stories
-   Functional Requirements
-   Sprint Backlog
-   Release Notes

------------------------------------------------------------------------

# Governance Rules

-   Every PCR must receive a unique identifier.
-   Significant changes require product approval.
-   Scope changes during an active Sprint should be avoided.
-   Implemented PCRs must appear in Release Notes.

------------------------------------------------------------------------

# Goal

Provide a transparent and standardized workflow for managing product
changes while preserving roadmap alignment, development quality, and
delivery predictability.
